Search Typeahead
System.

A search-autocomplete backend built around one idea: keep the read path off the database entirely. A prefix trie answers every keystroke in memory, a hand-written consistent-hash ring spreads cached results across three real Redis nodes, recency-aware ranking surfaces what is trending now, and writes are batched so popularity counts never touch SQLite one search at a time.

Stack: Node.js · Express · node:sqlite · Redis ×3 (Docker)  ·  Dataset: 149,803 Wikipedia page titles  ·  Ring: 150 virtual nodes/node

Architecture

The system deliberately splits reads (suggestions) from writes (search submissions). Reads are served from memory and the cache and never block on the database. Writes are buffered and applied to SQLite in aggregated batches. The trie and cache are derived, rebuildable views; SQLite is the single source of truth.

Consistent-hash routing

Why reads never hit SQLite

After the one-time startup load, suggestions come from the cache or the trie — both in memory. SQLite is touched only by the startup load and by batched writes. A 20,000-request benchmark produced exactly one database read. That is the entire purpose of the trie + cache layer.

Fault tolerance The cache is an optimization, not a hard dependency. If a Redis node is unreachable, the read degrades to a cache miss and is served from the trie instead of failing. Killing a node mid-traffic was verified to keep every request returning 200, with the cache resuming cleanly on recovery.
02

Dataset

Requirement: at least 100,000 queries, each with a count. Source: Wikipedia pageviews hourly dumps. Each line is domain page-title view-count bytes; filtering English traffic for a single hour yields hundreds of thousands of title → count rows. The page title becomes the query and the view count is the popularity signal.

Why this dataset: it is fully open (no privacy issues, unlike the withdrawn AOL query log), it carries a genuine count per entry, it is reproducible from a fixed URL, and one hour of English traffic comfortably clears the 100k minimum. The bundled data/queries.csv holds 149,803 distinct queries built this way.

Verified end to end Run against a live dump: 7.75M lines scanned → 456K candidate titles → clean query,count output — counts sorted descending, zero namespace junk, zero malformed rows.

Design choices & trade-offs

Stack

LayerChoiceReason
BackendExpressOne language across the stack; the trie/cache/ring logic reads cleanly in JS.
Datastorenode:sqliteOne file, real SQL, no native module to compile; synchronous API makes the read/write paths easy to reason about and to count.
CacheRedis ×3Three independent processes (ports 6379–6381); the app routes across them via a ring it implements itself. An in-memory backend mirrors the same ring for tests/offline runs.
FrontendVanilla JSNo build step; a debounced input and a dropdown are all the UI needs.

Trie for suggestions

A prefix trie gives O(prefix length) lookups, independent of dataset size, because every node caches the top-K completions in its subtree. The alternative — a sorted array with binary search — finds the prefix range in O(log n) but must then scan and sort that whole range per request, which is far worse for short, hot prefixes like "a" that match a huge slice of the data. A key invariant keeps it cheap: counts only ever increase, so top-K is updated incrementally on each write rather than recomputing a subtree.

Trade-offThe materialized top-K is memory-hungry (~679 MB RSS for 149,803 queries — see §5). That is the deliberate price of flat read latency, and it is tunable by lowering the candidate-pool size K.

Consistent hashing in application code

The ring lives in our code, not in a Redis cluster. A cluster would hash keys for us inside the client, hiding the exact mechanism the assignment asks us to implement. So we run three plain, independent Redis nodes and route ourselves: hash the prefix, walk the ring clockwise, pick the node. 150 virtual nodes per physical node plus a murmur3-style finalizer keep the load even — without virtual nodes a 3-node ring splits traffic very unequally.

Why it mattersAdding or removing a node remaps only ~1/N of keys versus nearly all of them for naïve hash(key) % N. Measured: adding a 4th node moved 26.7% of keys, where modulo would move about 75%.

Cache-aside with TTL

/suggest is cache-aside: check the cache, compute from the trie on a miss, write back under a short TTL (default 30s). The TTL bounds staleness, and writes additionally invalidate the affected prefixes on flush.

Trade-offSuggestions can be up to one flush interval (~2s) or one TTL stale — fine for typeahead, where freshness within seconds is plenty and read latency stays flat.

Batch writes

Writing to SQLite once per search would make the DB the bottleneck. Instead searches are buffered and duplicates aggregated, so 50 searches for one query become a single count += 50 update.

Failure modelThe buffer is in memory, so a crash loses at most one flush interval of searches — and only popularity counts, never user data. The trie is rebuilt from SQLite (the source of truth) on restart. Hardening would mean a write-ahead log before acking, reintroducing the synchronous write batching exists to avoid.

Trending: continuous exponential decay

Rather than discrete hourly buckets, each query keeps a single exponentially-decaying recency accumulator with a ~10-minute half-life. The suggestion score is log(1 + count) + W · decayed_recent_weight. A short spike decays to zero on its own, so it cannot permanently over-rank a query; decayed entries are pruned on each flush.

Availability over strict consistency

A cache-node failure degrades to a trie read rather than an error. We chose to keep suggestions working from the trie over surfacing a cache outage to the user — the right call for a best-effort suggestion feature.

Reads avoided, writes reduced, memory cost

  • Database reads: 1 total across the 20,000-request run (the startup load). The read path never touches SQL.
  • Write reduction: ~18× measured on repeated popular queries; the ratio scales with how repetitive and bursty traffic is (a controlled 60-search / 2-query test showed 30×).
  • Memory footprint: ~679 MB RSS with the full dataset loaded — the cost of the materialized per-node top-K that buys O(prefix) reads, tunable via the candidate-pool size K.
